# Hand-authored raw-SQL migrations

snekql adds versioned migrations: a name-ordered `dict[str, str]` of raw SQL
passed to `Database.initialize()`, applied exactly once and recorded one row per
name in a snekql-owned `snekql_migrations` table (the Migration History). This
reverses the previously documented "startup schema management, **not**
migrations" non-goal.

We chose **hand-authored** migrations over autogenerated ones to preserve
snekql's explicit, no-magic, SQL-shaped posture: the dangerous parts of
migration tooling — schema diffing and the SQLite rebuild-copy dance — are
exactly what snekql refuses to hide. Migrations are the **sole schema-creation
authority** while **drift verification stays on**, so models remain the enforced
contract a migration list must converge to. snekql does **not** wrap migration
bodies in its own transaction (the author owns transactions; bookkeeping is
non-atomic, so migrations should be idempotent), and v1 does **not** coordinate
concurrent runs — both deliberate "caller's responsibility" boundaries
consistent with the rest of the library.

## Considered Options

- **Autogenerate revisions from a model-vs-live diff (Alembic `--autogenerate`).**
  Rejected: it is the most magical, error-prone part of ORM-adjacent tooling and
  contradicts snekql's no-magic philosophy. (Not even left as a "both" door.)
- **snekql-native schema-change builder for migration bodies (`add_column`, …).**
  Deferred: raw SQL ships first; a typed builder may follow.
- **Single-head version pointer vs. per-row history.** Chose per-row keyed by
  name: once migrations are named, set-difference is the simplest model to reason
  about and dissolves the positional/append-only corruption risk.
- **snekql-owned transaction boundary; advisory locking for concurrency.**
  Both rejected for v1 in favor of caller responsibility; locking tracked
  separately.

## Consequences

- A model change with no matching migration fails startup under `strict` — the
  drift check is a correctness net on the migration list.
- A crash between a migration succeeding and its history row being written
  re-runs that migration; migrations must be idempotent.
- Concurrent `initialize()` with pending migrations is undefined on MariaDB in
  v1; migrate from a single place.
- `docs/schema-drift.md` must be updated when migrations ship.

## Amendments

The core of this ADR stands: migrations are hand-authored raw SQL, applied
exactly once, recorded per name in the Migration History, with bodies and
bookkeeping non-atomic (so bodies must be idempotent). Later ADRs change only
*where and when* migrations run, and add a narrow generation affordance:

- **Concurrency** — [ADR 0002](0002-advisory-locked-concurrent-migrations.md)
  reverses the "concurrent runs are undefined; migrate from a single place"
  boundary with an always-on advisory lock.
- **Entry point and timing** — [ADR 0007](0007-imperative-migrations-connect-only-initialization.md)
  removes migrations from `Database.initialize()`. They are now applied by an
  explicit `migrate(...)` verb on a connect-only `Database`, and **automatic
  table creation from models is removed entirely**: migrations become the
  unconditional, sole schema-creation path (the "sole schema-creation authority
  *while* drift verification stays on" framing above no longer has a second mode
  to contrast with).
- **Verification** — [ADR 0008](0008-separate-partial-schema-verification.md)
  moves drift verification out of startup into an explicit `verify(...)` verb and
  pins its scope as a partial, structural check.
- **Scaffolding** — the "never generates" stance is narrowed: snekql never
  **authors or applies** migrations autonomously, but it **may scaffold** the
  initial `CREATE TABLE` DDL for a model as text the author then owns. The
  scaffold output is hand-authored from the moment it is committed — append-only,
  immutable, identical in status to any other body. snekql still refuses the
  dangerous part it rejected here: model-vs-live **diffing** and the SQLite
  rebuild-copy dance. No `ALTER`-generation, no autogenerate.
